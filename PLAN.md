# Plan: Improve the susops codebase

_Locked via grill (interview skipped at user request) — by Claude + manuel.schmid_

## Goal

Raise correctness and maintainability of susops without adding features or speculative abstractions. The work is a **prioritized portfolio of four independently-shippable phases**, ordered by value-to-risk: fix real latent bugs that bite on the primary platform (macOS), delete confirmed-dead code, collapse one triplicated-and-mistested subsystem (browser launch), and close the highest-leverage test gaps. Each phase can ship alone; later phases are not prerequisites for earlier ones. Lazy lens throughout: prefer deletion over addition, reject speculative refactors. The single rejected proposal — splitting the well-sectioned 2541-LOC facade into mixins — is explicitly out of scope.

Findings come from four parallel codebase audits (over-engineering/dead-code, structural/testability, test-coverage, correctness/resource-leaks); each cited line was read before inclusion.

## Approach

### Phase 1 — Correctness fixes (highest value; real bugs on the primary platform)

Process model (verified): the facade (`SusOpsManager`) runs in the `susops-services` daemon; all frontends talk to it over RPC via `SusOpsClient` (`client.py`). RPC calls execute in **executor threads** (`rpc_server.py:98`), concurrently with the reconnect-monitor and auth-watcher threads. The SSH master is usually a child of the daemon — but **not always**: after a daemon restart the previous masters are orphaned (reparented to init/launchd) yet still live, and tests construct `SusOpsManager` in-process. So `is_running` must be correct for PIDs that are alive-but-not-our-child. This nuance drives 1.1 and Key decision A.

1.1 **macOS zombie master reports "running" forever → reconnect wedges.** `core/process.py:89-111`: `is_running()` reads `/proc/{pid}/status` to detect `State: Z`; on darwin (no `/proc`) the read raises `OSError`, the `except OSError: pass` swallows it, and a zombie returns `True`. Effect (verified): `_try_reconnect` (`facade.py:1494`) and `start()` (`facade.py:1211`) short-circuit on `is_tunnel_running(...) or is_socket_alive(...)`, so a dead-but-unreaped master is treated as live — the reconnect monitor never re-spawns and the user can't restart without an explicit `stop()`. Fix: make `is_running` decide the running verdict from `psutil` process **status** (ZOMBIE / NoSuchProcess → not running; any live status → running) — this is correct for non-child PIDs too. Reaping our own zombies stays best-effort via `os.waitpid(pid, os.WNOHANG)` wrapped in `try/except ChildProcessError`, and its result does **not** affect the verdict. Keep the Linux `/proc` fast path. **[IMPLEMENTATION SPLIT — see Risks]** The psutil-status verdict above shipped and fixes the zombie wedge. The socket-authoritative refinement below is **deferred to a tested follow-up**: it is coupled to the auth-deadline marker (the only safe way to tell a stale master from one mid-2FA), and that safety path cannot be exercised in this test environment. Treat **`is_socket_alive` as the authoritative "connected" signal** in `_try_reconnect`/`start` short-circuits — a process alive but with a dead control socket is stale/pending, not connected. To distinguish a legitimate startup/auth-prompt window from a stale master, add a small **per-tag auth-pending marker** (`_auth_deadline: dict[str, float]`, set to `now + AUTH_TIMEOUT` when `_spawn_auth_watcher` starts at `facade.py:1234`, cleared on the watcher's socket-up path at `facade.py:793` and on exit/timeout). `_start_times` (`facade.py:591`) is uptime state and is *not* a prompt signal, so this marker is new but tiny. It is touched from the watcher thread, the reconnect monitor, and RPC threads, so guard it under the per-tag lifecycle lock (or a tiny dedicated auth lock) — never read/write it unlocked. A process-alive/socket-dead tag reads as **pending** while within its `_auth_deadline`, and **stale** once the deadline passes or `ssh -O check` fails. **`_try_reconnect` interaction (important):** it currently spawns a master, waits 10 s, and returns `False` if the socket isn't up (`facade.py:1502-1513`) — without spawning the auth watcher or cleaning up. Under the socket-authoritative rule, that would re-spawn a fresh master on *every* poll while the first sits at a 2FA prompt → master pileup. Fix: reconnect must share the same pending-auth path as `start()` — if a master is already alive for the tag and still within its auth deadline, return without re-spawning (it's pending, not failed); if reconnect *does* spawn and the socket never comes up and the tag is not pending-auth, **stop the spawned master before returning `False`** so no orphan is left. The per-tag lifecycle lock (1.6) additionally prevents the monitor and a manual `start` from spawning concurrently. See Key decision A.

1.2 **`_share_servers` mutated cross-thread with no lock → `RuntimeError` on the poll path.** `facade.py`: `list_shares` iterates `self._share_servers.values()` **unwrapped** at line 2206 and `del`s entries at 2199-2201, while the auth-watcher thread (815/1103) inserts and RPC `stop`/`stop_share` (1607/2148) deletes. The dashboard/tray poll `list_shares` every ~2 s. A start or stop coinciding with a poll raises `RuntimeError: dictionary changed size during iteration`. Fix: add `self._share_lock = threading.Lock()`, held **only** around dict snapshot/pop/insert and released before any I/O (`server.stop()`, `start_forward`) or `_config_lock` acquisition — see Key decision B for the lock-ordering rule that avoids deadlock.

1.3 **Async server `stop()` nulls the runner even when cleanup failed → port/listener leak + permanent lie.** `core/share.py:230-234` and `core/status.py:178-182`: both do `future.result(timeout=5)` / `except Exception: pass`, then unconditionally `self._runner = None`. On a cleanup timeout the listener can keep the port bound. Naively leaving `_runner` set would instead make `is_running()` lie forever and let a retry double-schedule cleanup. Fix: keep the cleanup `future`, clear `_runner`/`_site` state **only** from a success callback on that future, log the failure with context on timeout, and block `start()` while a cleanup future is still pending (so neither a stale-True nor a double-schedule can occur).

1.4 **`find_master_pid` is a no-op on macOS → stale PID never recovered.** `core/ssh.py:295-314` scans `/proc`, returns `None` on darwin. Callers `_connection_status` (`facade.py:890`) and `_restore_pac` (`facade.py:1032`) then never refresh a stale PID file, so the dashboard shows `pid=None` and the bandwidth sampler can't map bytes to the tag. Degrades gracefully (socket-liveness still reports running) — cosmetic + sampling loss. Fix: add a darwin branch via `psutil.process_iter` that matches **strictly** — `name`/`exe` is `ssh`, exact `ControlPath=<socket>` in the cmdline, `ControlMaster=yes`, same UID — and confirm with `ssh -O check` before adopting, so it can never adopt an unrelated process. When a recovered PID is written back to its PID file, route it through `ProcessManager.track_existing` (1.5) so it carries `create_time` and isn't reuse-vulnerable. **Factor this matcher once and reuse it for `_sweep_orphan_masters` (`ssh.py:264`)**, which currently kills any `ssh` process whose argv merely *contains* the socket path as a **substring** (`sock_str in arg` → `proc.kill()`) — a loose match that can kill an unrelated process. Replacing the substring test with the strict matcher (exact `ControlPath`, same UID) closes that hole and keeps one source of truth. Lower priority within the phase, but the `_sweep_orphan_masters` substring→strict fix rides along with 1.5's identity work.

1.5 **PID-reuse identity + unconditional unlink + blind kill.** Three related issues, same class as the zombie bug: (a) `core/process.py:71-74` `stop()` always `unlink()`s the PID file even if SIGTERM didn't take, orphaning a surviving process (invisible to `kill_all`/`status_all`); `kill_all` (`process.py:130`) has the same unconditional unlink; (b) `stop()`/`kill_all` `os.kill(pid, sig)` a PID read from the file with **no identity check**; (c) more broadly, a reused PID also makes `is_running`/`is_tunnel_running` return `True` for an unrelated process, blocking restart. ssh/socat/pac processes share no uniform argv marker, so cmdline matching is unreliable — use **`create_time` identity** instead (the canonical PID-reuse defense): `ProcessManager.start` writes the PID file as `"<pid>:<create_time>"`; `is_running`, `is_tunnel_running`, `stop`, and `kill_all` read both and confirm the live process's `create_time` matches before trusting/​signalling the PID. `create_time()` comes free from the `psutil.Process(pid)` that Decision A already constructs. Mismatch → treat the entry as stale and unlink, don't signal. After SIGTERM, recheck liveness and escalate to SIGKILL if the process survives the wait loop; unlink only once exit is confirmed. Promoted to first-class Phase 1 — killing the wrong process, leaking live ones, and reuse-blocked restarts are all real.

  Four edges to close so the identity actually holds: **(a) start() race** — read `create_time()` immediately after `Popen`; if the child already exited / lookup raises `NoSuchProcess`, treat start as *failed* and write no PID file (the current `start` docstring already claims "Raises RuntimeError if process fails to start" but doesn't verify). **(b) adopted PIDs** — recovery paths (`find_master_pid` callers at `facade.py:890/1032`) currently write *raw* PIDs back, which would re-open the reuse window; add `ProcessManager.track_existing(name, pid)` that stamps `pid:create_time` and route every recovered/adopted PID through it. (Note: `_sweep_orphan_masters` is **not** a writer — it is a kill sweep; see 1.4.) **(c) legacy files** — a pre-existing `<pid>`-only file is reuse-vulnerable until rewritten; on the first successful liveness validation, upgrade it in place to `pid:create_time`; if validation is ambiguous *and* the socket is dead, unlink it as stale rather than trusting it. **(d)** keep a best-effort fallback only for the single poll before upgrade, so existing installs converge to the safe format on first contact.

1.6 **No per-connection lifecycle lock → check-then-act races.** RPC handlers run in executor threads (`rpc_server.py:98`) alongside the reconnect monitor and auth-watcher; `start`/`stop`/`restart`/`_try_reconnect`/`fetch` share no per-tag lock, and `ProcessManager.start` (`process.py:42`) blindly overwrites the PID file. The code already documents the hazard (`facade.py:579`: "parallel start() calls all see is_running()==False, each tries…"). A concurrent `start` + reconnect, or two `start`s, can spawn duplicate masters or overwrite a live PID file. Fix: add a per-connection `threading.Lock` (a `dict[str, Lock]` keyed by tag, created under a small guard lock) held across each connection's check→start/stop/reconnect **state transition and process mutation**. **Scope caveat for `fetch`:** a fetch can run a long download, so it must hold the lifecycle lock only around its master start/teardown transitions, **not** across the download body — otherwise `stop`/`remove_connection` block for the full fetch timeout. (Same "no blocking I/O under the lock" rule as the leaf locks.) Lock order (Key decision B): the per-tag lifecycle lock is **outermost**; `_config_lock`/`_share_lock` are leaf locks acquired *under* it only for a brief snapshot and released before any process I/O. **Prerequisite refactor:** `remove_connection` and `_remove_forward` currently hold `_config_lock` *across* process I/O (the reverse of the rule); rework them to snapshot the needed config under `_config_lock`, release it, then do the process/forward teardown under the per-tag lock. Without this they would violate the single lock order and risk the deadlock the order is meant to prevent.

1.7 **`socat` local-UDP EXEC string interpolates config values unsafely.** `core/socat.py:88-91`: `sock` is `shlex.quote`d but `conn.ssh_host` and `fw.dst_addr` are interpolated raw into the `ssh … socat - UDP4-SENDTO:{dst_addr}:{dst_port}` command that becomes `EXEC:'…'`. A host/address with a space or shell metacharacter breaks the forward and is a config-injection vector. Note: naive `shlex.quote` is **wrong** here — it emits single quotes that collide with socat's outer `EXEC:'…'` single-quote grammar (CLAUDE.md documents this exact fragility). Fix (lazier and sounder): **validate at the config trust boundary**, with two tiers since `ssh_host` and `dst_addr` have different grammars:
  - `ssh_host` is an OpenSSH destination — `user@host`, an `~/.ssh/config` alias, or an IP — so a strict hostname/IP regex would wrongly reject valid configs. Validate it as a **single shell-safe token**: reject whitespace and shell metacharacters (`; | & $ \` ' " < > ( ) \n` etc.), which defeats injection while allowing `user@host` and aliases.
  - `dst_addr`/`src_addr` are forward endpoints; the existing/stricter hostname-or-IP rule is appropriate there. Keep it compatible with the documented valid bind values (`localhost`, `172.17.0.1`, `0.0.0.0` — see CLAUDE.md) and ordinary host aliases; the goal is rejecting metacharacters/whitespace, not narrowing accepted addresses.

  This removes the unsafe-interpolation problem at the source rather than escaping into a nested socat-EXEC-shell grammar. Tests: config rejects an `ssh_host` and a `dst_addr` containing space, `;`, and `'`; config accepts `user@host` and a bare alias as `ssh_host`. (If arbitrary destinations must ever be supported, fall back to a wrapper argv/script — YAGNI for now.)

1.8 **PID/port files are world-readable (`0644`) while the config is already `0600`.** README.md:94-98 documents `chmod 600` on the port file as "the cheapest fix… not currently the default," and `config.py:209` already writes `config.yaml` at `0o600` — so the pid/port files are an inconsistency, not a deliberate choice. Fix: create `<workspace>/pids` as `0700` **and `chmod` it if it already exists** (a bare `mkdir(exist_ok=True)` won't relax an existing `0755`); write PID/port files **atomically** — write to a temp file, `chmod 0600`, then `os.replace` into place — `chmod`-ing any pre-existing `0644` files, mirroring `config.py:209-230`. Atomicity matters here, not just perms: `SusOpsClient` polls the port file every 100 ms (`client.ensure_daemon_running`), so a plain `O_TRUNC` write exposes a window where a reader sees an **empty** file and concludes the daemon/tunnel vanished. Factor one tiny `_atomic_write(path, text, mode=0o600)` helper and use it for both PID and port files. This is the *cheap* hardening only; full RPC authentication (bearer token / Unix socket) is explicitly **out of scope** — see Risks.

**Cross-cutting (applies to every new failure branch in Phase 1):** add structured `log.warning`/`log.debug` with PID + process name + tag + exception at each new swallow point — `psutil` lookup failures (1.1/1.4), cleanup timeouts (1.3), identity-mismatch skips and SIGKILL escalation (1.5), and the bandwidth sampler's existing exception handler. The current code mostly swallows these silently; a regression here is invisible without logs.

**Phase 1 acceptance criteria — each fix ships with its own regression test** (since phases are independently shippable, these belong to Phase 1, not Phase 4): zombie/non-child identity (1.1 — `psutil` status mocked: zombie→not-running, live non-child→running); `_try_reconnect` returns `False` for PID-alive/socket-dead, and does **not** re-spawn while a master is pending-auth (1.1); `create_time` mismatch → stale entry not signalled, and SIGTERM-ignored → SIGKILL escalation + unlink-only-after-exit (1.5); duplicate concurrent `start` prevented by the per-tag lock + `fetch` not blocking `stop` for its full timeout (1.6); aiohttp cleanup-timeout leaves `is_running()` truthful and blocks restart (1.3); strict matcher rejects a substring-only `ssh` process in the orphan sweep (1.4); pids dir/file mode is `0700`/`0600` after start incl. migration of a pre-existing `0644` file, and a concurrent reader never sees an empty port file (1.8 — atomic write); config rejects `ssh_host`/`dst_addr` with space/`;`/`'` and accepts `user@host` (1.7).

## Key decisions & tradeoffs

**A. Zombie detection: the verdict comes from `psutil` status; `waitpid` is best-effort reaping only.** `os.waitpid(pid, WNOHANG)` only works on direct children of the *calling* process, and `is_running` is NOT always called for a child PID — orphaned-but-live masters after a daemon restart, and in-process test fixtures, hand it non-child PIDs. The critical correction: a non-child PID must **not** be reported "not running" just because `waitpid` can't reap it — a live adopted master is genuinely running. So the running verdict is decided solely by `psutil.Process(pid).status()` (`ZOMBIE` or `NoSuchProcess` → not running; any live status → running), which is correct for children and non-children alike. Reaping our own zombies is a separate best-effort step (`waitpid(..., WNOHANG)` in `try/except ChildProcessError`) whose result is *ignored* for the verdict. Identity verification before *killing* (Decision separate, fix 1.5) prevents acting on a reused PID. Tradeoff: a `psutil` call on the non-Linux path (negligible). Rejected: Linux-only `waitpid` reaping (bug is on macOS), bare `os.kill(pid,0)` (returns 0 for zombies — the original bug), and the earlier draft's "non-child → not running" (wrong; would kill reconnect for adopted masters).

**B. Share-dict lock + a single global lock order.** A bare `list(...)` snapshot at `facade.py:2206` closes the immediate crash, but a dedicated `_share_lock` also protects the read-modify-write in `stop_share`/auth-watcher and matches the existing `_config_lock`/`_pac_start_lock` pattern. The hazard a lock introduces is deadlock if held across I/O or another lock. **Single global lock order:** per-tag lifecycle lock (1.6) is outermost; `_config_lock` and `_share_lock` are leaf locks acquired *under* it (never the reverse) and held only for a snapshot/pop/insert, then released **before** any blocking I/O (`server.stop()`, `start_forward`, subprocess, network). The two leaf locks are never held simultaneously. This is consistent with 1.6's "no process I/O under a leaf lock" — acquiring `_config_lock` briefly under the lifecycle lock for a snapshot is allowed; *holding* it across I/O is not (the `remove_connection`/`_remove_forward` refactor in 1.6 exists precisely to fix the current violation). Tradeoff: more locks to reason about; mitigated by the one documented order and tiny critical sections.

**C. Extract only `_BandwidthSampler`; reject the facade-into-mixins split.** The facade is large (2541 LOC) but cleanly sectioned (`# Lifecycle`/`# PAC hosts`/`# Port forwards`/`# File sharing`/`# Testing` dividers). Splitting cohesive method clusters into mixins is speculative complexity with no testability payoff and is explicitly rejected. The bandwidth sampler is the one piece that is already a standalone class with an injected dependency, so extraction is near-zero-risk and unlocks isolated parser tests. `_ReconnectMonitor` is a maybe-later (it calls back into four manager methods, so extraction needs an interface seam) — out of scope here.

**D. Phases are independently shippable and independently reviewable.** This is a portfolio, not one change. Phase 1 (bugs) delivers value alone; Phase 2 (deletions) is pure subtraction; Phase 3 needs Phase 2's `base` deletions but nothing else; Phase 4 is additive. Recommended order is 1 → 2 → 3 → 4, but the user may ship any subset.

## Risks / open questions

- **Dead-code verification is the deletion gate.** grep-clean ≠ safe-to-delete (the `test_ssh`/`share_is_running` RPC trap proves it). Before any Phase 2 deletion, a verification pass must dump the actual `rg` output proving zero callers AND confirm each symbol is absent from `_ALLOWED_METHODS`, any `getattr`/dynamic dispatch, and PyObjC selector-string registration. A wrong "it's dead" ships as a regression. (Done: per-symbol `rg` evidence collected and folded into Phase 2; two corrections applied — keep `FWD_PROCESS_PREFIX`, and `_write_pid_file` is dead-by-supersession not dead-by-absence.)
- **`is_running` is called ~25 times across the facade hot path.** The zombie fix changes its return value for a class of PIDs; every caller currently treats `True` as "leave it alone." Need to confirm no caller breaks when a previously-"running" zombie now returns `False` (the intended behavior is that the reconnect monitor re-spawns — verify that path is correct, not just unblocked).
- **macOS-only behavior is hard to unit-test on CI.** Fixes 1.1/1.4 touch darwin branches; CI may be Linux. The new tests must mock `psutil`/platform rather than depend on the host OS, or be `skipif`-gated honestly (the audit found the existing zombie test silently no-ops on macOS).
- **GUI/AppKit code (`mac_config_window.py` 4044 LOC, `mac.py` 3413) stays integration-tested only.** Unit-testing AppKit window behavior is an inherent limit, not a gap this plan tries to close.
- **RPC trust model is deliberately single-user — full authentication is deferred, not forgotten.** The 127.0.0.1 `/rpc` endpoint (`rpc_server.py:121`) has no auth, and `list_shares` returns `ShareInfo.password` (`types.py:170`) in cleartext to any local caller. README.md:98 documents this as an accepted "single-user trust zone." Adding a per-daemon bearer token / Unix-socket auth (README.md:95-96) is a real feature but is scope creep on a documented design decision; this plan does the *cheap* half (0600/0700 file perms, fix 1.8) and leaves bearer-token auth as a flagged, separate decision for the owner. If the threat model includes co-resident hostile users, that work should be its own task.
- **DEFERRED (during implementation): socket-authoritative reconnect + auth-deadline marker.** The primary bug (macOS zombie wedge) is fixed and committed via the psutil-status verdict in 1.1, and the per-tag start lock (1.6) shipped. The socket-authoritative reconnect refinement + `_auth_deadline` marker are a single coupled bundle (without the marker you either don't fix the hung-master case or you regress by killing masters mid-2FA), and the marker's safety-critical path — don't kill mid-auth — cannot be verified in this environment (no real auth prompts). Shipping it here would land its most dangerous behavior unverified, so it is split into a separate, properly-tested follow-up. The residual it addresses (a non-zombie ssh master alive with a dead control socket) is rarer than the zombie case already fixed. This is the materialization of the startup-window risk flagged below — surfaced to the user, not silently dropped.
- **`is_socket_alive`-as-authoritative (1.1) must not break the startup window.** During a fresh `start`, the master process is alive before its control socket binds. Treating "process alive, socket dead" as not-connected is correct for reconnect/stale detection but must not cause re-spawn churn during normal startup — the fix must scope the socket-authoritative check to the reconnect/short-circuit paths and keep the existing wait-for-socket grace on initial start.

## Out of scope

- Splitting the facade into mixins or per-concern modules (rejected — Key decision C).
- Extracting `_ReconnectMonitor` (needs an interface seam; revisit only if it later earns its keep).
- The `time.sleep` polling-seam refactor (`facade.py:1482/1512/1651/2281`) — only if the suite is measurably slow; defer.
- A `manager.list_ssh_hosts()` passthrough for `connections.py:23`'s direct `get_ssh_hosts` import — low-priority nice-to-have, not a correctness issue.
- Full RPC authentication (per-daemon bearer token / Unix-socket auth) — deliberate single-user trust model; deferred to its own task (see Risks). The cheap file-perm half is in 1.8.
- Any new feature, new dependency, or new frontend capability.
- Reworking the socat command-string tests (they guard documented bugs — keep).
